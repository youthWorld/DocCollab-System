# 配置管理计划

> _本配置管理计划按CMMI规范制定，使用GitHub进行版本控制。_

---

## 一、引言

### 1.1 目的

本文档定义"多人文档协作系统（DocCollab）"项目的配置管理策略，确保项目在开发过程中所有配置项（代码、文档、资源等）的版本可控制、变更有记录、状态可追溯。

### 1.2 范围

本计划适用于DocCollab项目从v1.0到v2.0的全生命周期，涵盖所有源代码、需求文档、设计文档、测试用例等配置项。

### 1.3 参考标准

- CMMI-DEV V2.0 配置管理（CM）过程域

## 二、版本控制平台

### 2.1 平台选择

| 属性     | 内容                                               |
| -------- | -------------------------------------------------- |
| 平台     | GitHub                                             |
| 仓库地址 | https://github.com/youthWorld/DocCollab-System.git |
| 主分支   | main                                               |
| 访问权限 | 公开仓库                                           |

### 2.2 分支策略

本项目为单人开发项目，采用简化的分支策略：

- **main**：主分支，保存稳定、可发布的代码
- 各版本开发直接在main分支上迭代提交
- 每个版本里程碑完成后打Tag标记

---

## 三、版本标签规划

### 3.1 版本命名规则

版本号格式：`v<主版本号>.<次版本号>`

- **主版本号**（Major）：重大功能变更或架构调整时递增（如v1 → v2）
- **次版本号**（Minor）：功能迭代递增（如v1.0 → v1.1）

### 3.2 发布标签

| 标签名称     | 对应版本 | 计划日期 | 说明                                               |
| ------------ | -------- | -------- | -------------------------------------------------- |
| v1.0-release | v1.0     | 第1周末  | 核心功能：版本容器管理、文档协作编辑、资源上传下载 |
| v1.1-release | v1.1     | 第2周末  | 迭代功能：修改标签与描述、变更记录                 |
| v1.2-release | v1.2     | 第3周末  | 迭代功能：版本对比                                 |
| v2.0-release | v2.0     | 第4周末  | 完善功能：容器分支、批量导出、容器标签             |

### 3.3 开发中标签

开发过程中可使用以下临时标签：

- `vX.X-beta`：功能开发完成，进入测试阶段
- `vX.X-rc`：发布候选版本

## 五、变更控制流程

### 5.1 变更控制流程图

```
变更申请 → 变更评估 → 变更审批 → 变更实施 → 变更验证 → 变更关闭
```

### 5.2 流程说明

| 步骤        | 角色       | 活动                             | 输出         |
| ----------- | ---------- | -------------------------------- | ------------ |
| 1. 变更申请 | 任何人     | 提交变更请求，说明变更原因和内容 | 变更请求记录 |
| 2. 变更评估 | 项目经理   | 评估变更影响范围、工作量、风险   | 评估报告     |
| 3. 变更审批 | 项目经理   | 审批变更请求（批准/拒绝/延期）   | 审批意见     |
| 4. 变更实施 | 开发工程师 | 在代码中实施变更，提交commit     | 代码修改     |
| 5. 变更验证 | 测试工程师 | 验证变更正确性，回归测试         | 验证结果     |
| 6. 变更关闭 | 配置管理员 | 更新配置项状态，关闭变更请求     | 更新配置状态 |

### 5.3 变更级别定义

| 级别         | 说明                   | 审批要求     |
| ------------ | ---------------------- | ------------ |
| 一级（重大） | 影响架构、新增核心功能 | 需正式评审   |
| 二级（普通） | 功能迭代、Bug修复      | 项目经理审批 |
| 三级（微小） | 文档修正、样式调整     | 自行记录     |

## 六、配置状态报告

### 6.1 报告频率

- 每个版本迭代结束时生成配置状态报告
- 配置状态报告包含：配置项清单、变更记录、版本基线状态

### 6.2 审计计划

| 审计节点    | 审计内容             |
| ----------- | -------------------- |
| v1.0 发布前 | 审计代码与需求一致性 |
| v1.1 发布前 | 审计变更记录完整性   |
| v1.2 发布前 | 审计版本标签正确性   |
| v2.0 发布前 | 全面审计所有配置项   |
