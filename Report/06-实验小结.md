# 实验小结

## 一、对CMMI的理解与体会

### 1.1 需求管理（REQM）

通过本次实验，我深刻理解了CMMI需求管理过程域的核心思想——**需求不仅仅是"写下来"，更要"管起来"**。

在编写需求规格说明书（SRS）的过程中，我实践了以下需求管理活动：

- **需求获取**：从课程小组实际协作的痛点出发，识别出"版本混乱""修改不可追溯""多类型文件分散"等真实需求。
- **需求文档化**：将需求按R1-R9编号，每个需求明确定义输入、处理、输出和约束，确保需求的完整性和可验证性。
- **需求分级**：按优先级将需求分为"必须有"（v1.0）、"迭代加入"（v1.1/v1.2）、"优化增强"（v2.0）三个层次，体现增量开发理念。
- **需求追溯**：通过需求编号（R1-R9），确保每个功能需求都能追溯到对应的开发任务和测试用例。

**体会**：需求管理的关键在于"双向追溯"——正向可从需求追踪到实现，反向可从变更追踪到需求来源。这是确保软件质量和应对变更的基础。

### 1.2 配置管理（CM）

配置管理是本次实验收获最大的实践领域。通过建立完整的配置管理体系，我认识到：

- **版本控制不只是"保存代码"**：通过GitHub的Tag机制，我为v1.0到v2.0规划了清晰的版本标签（v1.0-release等），使每个里程碑都有明确的版本基线。
- **配置项要全面**：不仅代码是配置项，文档（6份报告）、测试用例等也属于配置项，需要统一管理。
- **变更控制要规范**：我设计了三级变更控制流程（重大/普通/微小），确保每次变更都有记录、有审批、有验证。

**体会**：配置管理的本质是"在正确的时间使用正确的版本"。没有配置管理，项目就会陷入"哪个版本是最新的？""这个Bug是谁引入的？"的无尽混乱。

### 1.3 项目计划（PP/PMC）

项目进度计划的制定让我实践了CMMI项目规划（PP）和项目监控（PMC）的理念：

- **WBS分解**：将整个项目从6个阶段分解到22个具体任务，每个任务有明确的工作量估算和交付物。
- **里程碑驱动**：设置M1-M4四个里程碑，每个里程碑有明确的验收标准。
- **风险管理**：识别了5项风险，制定了缓解和应对策略。

**体会**：计划不是"写出来就放一边"的文档，而是指导执行和衡量偏差的基准。CMMI强调的计划与实际的比对（计划偏差分析），是项目管理最核心的活动。

## 二、遇到的问题及解决方法

### 问题1：如何在一人项目中体现"团队协作"？

**问题描述**：本项目为单人完成，但CMMI框架强调团队角色分工，如何在一人项目中合理体现？

**解决方法**：

- 按CMMI角色划分（PM、RA、CM、DEV、QA）定义职责，所有角色执行者写"本人"
- 通过Git的提交记录模拟多人协作的版本演进
- 在每个阶段结束后切换角色视角进行自检（如以QA视角检查开发质量）
- 在分工文档中明确"一人可承担多角色"的原则

### 问题2：如何平衡系统复杂度与实验要求？

**问题描述**：需求文档中提到技术栈包括Vue3、WebSocket、Yjs等，但作为演示系统，不应过度复杂化。

**解决方法**：

- 明确系统定位为"轻量级小型系统"，而非大中型生产系统
- 采用浏览器LocalStorage/IndexedDB作为存储方案，避免引入数据库和后端服务
- 将功能严格限制在需求文档列出的R1-R9范围内，不做额外扩展
- 在文档中强调"定位为小型演示系统"

### 问题3：版本命名规则的确定

**问题描述**：需求文档中版本规划为v1.0 → v1.1 → v1.2 → v2.0，其中v1.2到v2.0的跳跃较大，需要合理的命名解释。

**解决方法**：

- 采用`v<主版本>.<次版本>`的命名规范
- 解释v1.2到v2.0的跳跃：v2.0引入了架构层面的新特性（分支机制），属于重大功能增强
- 在配置管理计划中明确版本命名规则

## 三、不足与改进方向

### 3.1 存在的不足

1. **需求分析深度有限**：由于是一人独立完成，需求分析缺乏多方评审，可能存在需求遗漏或不合理的地方。
2. **测试覆盖不充分**：受时间限制，测试仅覆盖基本功能路径，边界测试和异常测试不够全面。
3. **配置管理实践简化**：实际GitHub上的提交和标签管理可能不够规范，与计划存在偏差。
4. **进度估算偏乐观**：17人天的工作量在4周内完成对单人来说比较紧张，实际执行可能需要调整。
5. **技术方案未充分验证**：富文本编辑器和协作编辑方案在实验阶段未完全实现和验证。

### 3.2 改进方向

1. **引入同行评审**：在真实项目中，需求文档应经过多方评审，确保需求的完整性和正确性。
2. **增强自动化测试**：引入单元测试和端到端测试框架，提高测试覆盖率和效率。
3. **严格遵循Git工作流**：在实际开发中严格按照配置管理计划执行commit规范和标签管理。
4. **建立度量体系**：引入进度偏差率、缺陷密度等量化指标，更科学地管理项目状态。
5. **持续集成**：在条件允许时引入CI/CD流水线，自动化构建和部署流程。

## 四、总结

本次实验通过设计"多人文档协作系统"项目，完整实践了CMMI框架下的需求管理、配置管理和项目计划三大过程域。虽然项目为单人完成且系统定位为小型演示，但通过撰写6份规范文档，我对软件过程管理的核心理念有了更深刻的理解：

- **需求管理**教会我：需求要有编号、可追溯、分优先级
- **配置管理**教会我：版本要打标签、变更有流程、基线要审计
- **项目计划**教会我：任务要分解、里程碑要明确、风险要管理

这些理念不仅适用于本次实验，更将指导我未来的软件工程实践。CMMI不是僵化的教条，而是一套经过验证的最佳实践框架——在实际项目中，可以根据项目规模和团队情况适当裁剪，但其核心思想（可追溯、可控制、可度量）永远不会过时。
