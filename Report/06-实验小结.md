# 实验小结

## 一、对CMMI的理解与体会

### 1.1 需求管理（REQM）

通过本次实验，我深刻理解了CMMI需求管理过程域的核心思想——**需求不仅仅是"写下来"，更要"管起来"**。

在编写需求规格说明书（SRS）的过程中，我实践了以下需求管理活动：

- **需求获取**：从课程小组实际协作的痛点出发，识别出"版本混乱""修改不可追溯""多类型文件分散"等真实需求。
- **需求文档化**：将需求按 R1-R8 编号，每个需求明确定义输入、处理、输出和约束，确保需求的完整性和可验证性。
- **需求分级**：按优先级将需求分为 v1.0 核心功能（R1-R6）和 v2.0 完善功能（R7-R8）两个层次，体现增量开发理念。
- **需求追溯**：通过需求编号（R1-R8），确保每个功能需求都能追溯到对应的开发任务和测试用例。

**体会**：需求管理的关键在于"双向追溯"——正向可从需求追踪到实现，反向可从变更追踪到需求来源。这是确保软件质量和应对变更的基础。

### 1.2 配置管理（CM）

配置管理是本次实验收获最大的实践领域。通过建立完整的配置管理体系，我认识到：

- **版本控制不只是"保存代码"**：通过 GitHub 的 Tag 机制，为 v1.0 和 v2.0 规划了清晰的版本标签（v1.0-release 等），使每个里程碑都有明确的版本基线。
- **配置项要全面**：不仅代码是配置项，文档（6份报告）、测试用例等也属于配置项，需要统一管理。
- **变更控制要规范**：设计了三级变更控制流程（重大/普通/微小），确保每次变更都有记录、有审批、有验证。

**体会**：配置管理的本质是"在正确的时间使用正确的版本"。没有配置管理，项目就会陷入"哪个版本是最新的？""这个 Bug 是谁引入的？"的无尽混乱。

### 1.3 项目计划（PP/PMC）

项目进度计划的制定让我实践了CMMI项目规划（PP）和项目监控（PMC）的理念：

- **WBS分解**：将整个项目从4个阶段分解到18个具体任务，每个任务有明确的工作量估算和交付物。
- **里程碑驱动**：设置 M1-M2 两个里程碑，每个里程碑有明确的验收标准。
- **风险管理**：识别了5项风险，制定了缓解和应对策略。

**体会**：计划不是"写出来就放一边"的文档，而是指导执行和衡量偏差的基准。实际项目中，版本标签名称、协作方案等在开发过程中根据实际情况进行了调整，这正是 CMMI 强调的"计划 vs 实际"比对的意义所在。

## 二、系统实现概况

### 2.1 技术栈

| 层级         | 技术选型                                 |
| ------------ | ---------------------------------------- |
| 前端框架     | Vue 3 + TypeScript                       |
| 构建工具     | Vite                                     |
| UI 组件库    | Element Plus                             |
| 富文本编辑器 | TipTap（StarterKit + Underline）         |
| 状态管理     | Pinia                                    |
| 路由         | Vue Router（Hash 模式）                  |
| 数据存储     | LocalStorage                             |
| 协作方案     | BroadcastChannel API（多标签页实时同步） |
| 导出功能     | JSZip + Turndown（HTML → Markdown）      |

### 2.2 实现功能清单

| 编号 | 功能           | 说明                                                          |
| ---- | -------------- | ------------------------------------------------------------- |
| R1   | 版本容器管理   | 创建项目自动生成 v1.0 初始工作区，支持保存新版本              |
| R2   | 文档协作编辑   | TipTap 富文本编辑器，BroadcastChannel 多标签页实时同步        |
| R3   | 资源上传下载   | 文件上传转 base64 存储，支持一键下载                          |
| R4   | 版本标签与说明 | 5 种里程碑标签（初始版本/功能迭代/Bug修复/文档完善/代码重构） |
| R5   | 变更记录看板   | 每个版本卡片展示，展开查看变更明细、文件/文档快照             |
| R6   | 版本对比       | 选择两个版本比对新/删/改的文件和文档                          |
| R7   | 批量导出       | JSZip 打包，文档自动转 Markdown                               |
| R8   | 容器标签       | 自定义标签标记重要版本                                        |

### 2.3 核心设计决策

1. **版本标签从操作视角改为里程碑视角**：原设计中标签为"添加文件/删除文件/修改文件/修改文档"，实践中发现操作型标签语义模糊——一个版本可能同时包含多种操作。改为"初始版本/功能迭代/Bug修复/文档完善/代码重构"更能描述版本的**目的**。

2. **协作方案简化**：原计划使用 WebSocket + Yjs/OT 方案，但在实际开发中，考虑到系统定位为小型演示项目以及"无需后端服务器"的约束，改用浏览器原生 BroadcastChannel API。同一浏览器打开多个标签页，每个标签页作为一个独立用户参与协作，满足演示需求。

3. **版本语义修正**：初版实现中最新版本同时作为"工作区"被修改，导致版本之间差异计算不准。修正后所有编辑写入工作区（最新版本），"保存新版本"操作将当前工作区冻结为新版本。

## 三、遇到的问题及解决方法

### 问题1：版本标签命名不直观

**问题描述**：最初使用"添加文件""删除文件"等操作型标签，用户在保存版本时难以选择（一个版本可能同时添加和修改了文件）。

**解决方法**：改为里程碑式标签（初始版本/功能迭代/Bug修复/文档完善/代码重构），让用户描述本版本的**目的**而非操作。

### 问题2：协作编辑的回音和不同步

**问题描述**：基于 BroadcastChannel 的协作方案中，存在编辑回音（自己的编辑广播回来再次触发）和修改不同步的问题。

**解决方法**：改用版本号去重机制（`remoteVersion` / `localVersion`），配合 TipTap 的 `onUpdate` 回调替代 `watch`，确保编辑链路清晰分离。

### 问题3：版本对比误报无差异

**问题描述**：版本对比功能只检测文档内容修改，遗漏了新增文档和删除文档的检测。

**解决方法**：在 `compareContainers` 函数中补全 `addedDocs` 和 `removedDocs` 逻辑。

## 四、不足与改进方向

1. **真正的多人协作**：当前使用 BroadcastChannel 仅支持同浏览器多标签页协作。要支持跨设备协作需要引入 WebSocket 服务器和 OT/CRDT 算法（如 Yjs）。
2. **数据持久化**：LocalStorage 存在容量限制和安全风险，生产环境应迁移至 IndexedDB 或后端数据库。
3. **测试覆盖不充分**：受时间限制，系统缺乏自动化测试。
4. **UI/UX 可进一步打磨**：如文档列表的拖拽排序、批量操作等功能。

---

## 五、总结

本次实验通过设计并实现"多人文档协作系统"项目，完整实践了 CMMI 框架下的需求管理、配置管理和项目计划三大过程域。从 6 份规范文档的编写到 8 项核心功能的实现，对软件过程管理的核心理念有了更深刻的理解：

- **需求管理**：需求要有编号、可追溯、分优先级
- **配置管理**：版本要打标签、变更有流程、基线要审计
- **项目计划**：任务要分解、里程碑要明确、风险要管理

CMMI 不是僵化的教条，而是一套经过验证的最佳实践框架。在实际项目中，可以根据项目规模和团队情况适当裁剪，但其核心思想——**可追溯、可控制、可度量**——永远不会过时。
