# 小组成员分工说明

> _本分工按角色职责划分，实际执行中一人可承担多角色。_

---

## 一、引言

本文档说明"多人文档协作系统（DocCollab）"项目的团队角色分工。由于本项目为研究生课程个人实验项目，所有角色均由**本人**承担，但按CMMI框架要求对各角色的职责进行明确划分。

---

## 二、角色与职责

### 2.1 角色分配总览

| 角色              | 执行者 | 职责概述                         |
| ----------------- | ------ | -------------------------------- |
| 项目经理（PM）    | 本人   | 项目整体管理、进度控制、风险管理 |
| 需求分析员（RA）  | 本人   | 需求获取、需求分析、SRS编写      |
| 配置管理员（CM）  | 本人   | 版本控制、配置项管理、基线发布   |
| 开发工程师（DEV） | 本人   | 系统架构设计、前端开发、功能实现 |
| 测试工程师（QA）  | 本人   | 测试用例编写、功能测试、缺陷跟踪 |

---

### 2.2 各角色详细职责

#### 项目经理（PM）

| 职责项   | 具体内容                                     |
| -------- | -------------------------------------------- |
| 项目规划 | 制定项目进度计划、WBS分解、里程碑定义        |
| 进度控制 | 跟踪各阶段任务完成情况，确保按计划推进       |
| 风险管理 | 识别项目风险，制定应对策略，定期评估风险状态 |
| 沟通协调 | 与课程老师沟通需求，确保项目方向正确         |
| 交付管理 | 组织各阶段交付物评审，确保交付质量           |

#### 需求分析员（RA）

| 职责项   | 具体内容                                     |
| -------- | -------------------------------------------- |
| 需求获取 | 分析课程项目中存在的协作痛点，明确用户需求   |
| 竞品分析 | 调研同类产品（微信群、网盘等），对比功能差异 |
| 需求编写 | 编写需求规格说明书（SRS），定义R1-R9功能需求 |
| 需求管理 | 建立需求追溯矩阵，管理需求变更               |

#### 配置管理员（CM）

| 职责项     | 具体内容                                       |
| ---------- | ---------------------------------------------- |
| 版本控制   | 管理GitHub仓库，维护分支和标签策略             |
| 配置项管理 | 维护配置项清单，管理文档和代码的版本状态       |
| 基线发布   | 在每个里程碑节点创建版本基线（v1.0-release等） |
| 变更控制   | 执行变更控制流程，记录变更历史                 |
| 配置审计   | 定期审计配置项完整性和一致性                   |

#### 开发工程师（DEV）

| 职责项   | 具体内容                                                       |
| -------- | -------------------------------------------------------------- |
| 技术选型 | 选择前端框架（Vue3）、UI组件库（Element Plus）、富文本编辑器等 |
| 架构设计 | 设计系统前端架构、数据存储方案                                 |
| 功能开发 | 按版本计划实现R1-R9全部功能                                    |
| 代码提交 | 编写规范的commit message，按功能模块提交代码                   |
| 技术文档 | 编写代码注释和必要的技术说明                                   |

#### 测试工程师（QA）

| 职责项   | 具体内容                       |
| -------- | ------------------------------ |
| 测试计划 | 制定各版本测试范围和测试策略   |
| 用例编写 | 编写功能测试用例               |
| 功能测试 | 在每个版本发布前执行功能测试   |
| 缺陷管理 | 记录和跟踪发现的缺陷，验证修复 |
| 测试报告 | 输出各版本测试报告             |

## 四、协作机制

由于本项目为单人完成，协作机制简化为：

1. **自检自查**：每个阶段完成后，切换角色视角进行自检（如：以测试工程师视角检查开发质量）
2. **文档驱动**：严格按照文档规范进行开发，确保需求→设计→开发→测试的可追溯性
3. **版本控制**：通过Git提交记录体现"多人协作"的版本演进过程
4. **时间管理**：按角色分配独立的时间块，避免角色混淆
