# 多人文档协作系统 - 完整需求说明

## 一、项目基本信息

- **项目名称**：多人文档协作系统
- **选题定位**：面向课程小组项目的轻量级协作管理工具，以“版本容器”为核心
- **项目周期**：4周（模拟迭代开发）
- **版本规划**：v1.0 → v1.1 → v1.2 → v2.0

---

## 二、需求分级（按优先级 + 版本）

### 🔴 v1.0 核心功能（必须有）

| 编号 | 功能         | 说明                                               |
| ---- | ------------ | -------------------------------------------------- |
| R1   | 版本容器管理 | 一个容器代表一次完整项目状态，包含文档、代码、资源 |
| R2   | 文档协作编辑 | 富文本编辑器，多人同时编辑同一篇文档               |
| R3   | 资源上传下载 | 代码、视频、图片等文件的上传和下载                 |

### 🟡 v1.1 / v1.2 普通功能（迭代加入）

| 编号 | 功能           | 说明                                             | 计划版本 |
| ---- | -------------- | ------------------------------------------------ | -------- |
| R4   | 修改标签与描述 | 每次修改必须填写标签（需求变更/Bug修复等）和描述 | v1.1     |
| R5   | 变更记录       | 查看容器内所有变更历史                           | v1.1     |
| R6   | 版本对比       | 对比两个版本的差异                               | v1.2     |

### 🟢 v2.0 完善功能（优化增强）

| 编号 | 功能     | 说明                               | 计划版本 |
| ---- | -------- | ---------------------------------- | -------- |
| R7   | 容器分支 | 支持从历史容器创建分支，独立演进   | v2.0     |
| R8   | 批量导出 | 一键导出容器内所有文件             | v2.0     |
| R9   | 容器标签 | 为重要容器打标签（如“最终提交版”） | v2.0     |

## 三、需要生成的文档清单（共6份）

### 文档1：选题与竞品分析报告

**需要包含的内容**：

- 选题来源与缘由（课程小组项目版本混乱、修改不可追溯、多类型产出物分散）
- 对比2个同类产品：微信群/QQ群文件传输、网盘（百度网盘/OneDrive）
- 对比维度：统一管理、版本管理、修改说明、文档协作、变更追溯
- 功能缺陷分析：无版本管理、无法追溯修改、无协作编辑
- 创新点：版本容器、强制修改标签、轻量级协作
- 拟定解决的关键问题：版本管理、变更追溯、多类型产出物统一管理

### 文档2：需求规格说明书（SRS）

**需要包含的内容**：

- 功能列表：按上述 R1-R9 完整列出，每个功能标注版本号
- 非功能需求：响应速度、并发支持、数据安全、易用性
- 需求变更记录（留表格模板）

### 文档3：配置管理计划

**需要包含的内容**：

- 版本控制平台：GitHub，提交到main分支即可，链接：https://github.com/youthWorld/DocCollab-System.git，我会完成本地和远程仓库的连接
- 版本标签规划：v1.0-release、v1.1-release、v1.2-release、v2.0-release
- 版本命名规则说明
- 配置项清单
- 变更控制流程

> 请在文档开头加一句话：*本配置管理计划按CMMI规范制定，使用GitHub进行版本控制。*

### 文档4：项目进度计划

**需要包含的内容**：
- WBS分解（按版本阶段）
- 甘特图（用表格形式呈现）
- 里程碑：M1(v1.0) / M2(v1.1) / M3(v1.2) / M4(v2.0)
- 风险计划

> 请在文档开头加一句话：*本计划按CMMI规范制定，为项目执行的指导框架，实际执行可根据情况裁剪。*

### 文档5：小组成员分工说明

**需要包含的内容**：
- 角色与职责：项目经理、需求分析员、配置管理员、开发工程师、测试工程师
- 所有角色执行者写“本人”

> 请在文档开头加一句话：*本分工按角色职责划分，实际执行中一人可承担多角色。*

### 文档6：实验小结

**需要包含的内容**：
- 对CMMI需求管理、配置管理、项目计划的理解
- 遇到的问题及解决方法
- 不足与改进方向

## 五、补充强调说明

1. 这是一个研究生课程实验，生成的各文档目的是体现软件过程管理思想（CMMI框架）
2. 文档要体现：需求可追溯、版本可控、计划有里程碑、分工有角色
3. 一人完成项目是允许的
4. 本文档提供的需求说明可作为基础，具体细节可自行补充
5. **重点**：不要把这个项目写得太复杂，原定位为大中型系统，但现在仅作为演示，所以定位为小型系统，实现需求中的功能即可，不用拓展内容（比如使用阿里云OSS实现持久化）
